## Цель Фазы 3

Превратить MVP в production-ready приложение:
- Стабильная работа камеры/микрофона/экрана на macOS и Windows
- Реальный чат, синхронизированный с веб-платформой

- Подписанные установщики

## 3.1. Критические исправления

### 3.1.1. MediaDevices API на macOS (ПРИОРИТЕТ 1)

**Проблема:** WKWebView на macOS не всегда предоставляет `navigator.mediaDevices`.

**Решение:**
- Перейти на нативный захват медиа через Rust (AVFoundation на macOS, Media Foundation на Windows)
- Создать Tauri commands: `start_camera`, `stop_camera`, `start_microphone`, `list_devices`
- Фронтенд получает видеопоток через IPC, а не через браузерный API
- Библиотеки: `nokhwa` (Rust, кроссплатформенная камера) или прямые FFI-вызовы к AVFoundation

**Критерий готовности:**
- Камера и микрофон работают на macOS Ventura+ и Windows 10+ без ошибок
- Не зависит от WKWebView capabilities

### 3.1.2. Захват экрана на macOS (ПРИОРИТЕТ 1)

**Проблема:** `getDisplayMedia()` недоступен в WKWebView.

**Решение:**
- Нативный захват через `ScreenCaptureKit` (macOS 12.3+) из Rust
- На Windows — через `win-capture` / DXGI Desktop Duplication
- Tauri command: `start_screen_capture(source_id)` → возвращает поток кадров
- Список окон/мониторов через нативный API

**Критерий готовности:**
- Пользователь видит список окон и мониторов в UI
- Захват работает без браузерных ограничений

### 3.1.3. Стабильность LiveKit подключения

**Решение:**
- Автореконнект при обрыве (exponential backoff)
- Индикатор качества соединения в UI (bitrate, packet loss, latency)
- Graceful degradation: снижение качества при плохом канале
- Таймаут подключения с понятным сообщением

## 3.2. Реальный чат

### 3.2.1. LiveKit Data Channel

**Текущее состояние:** Mock (локальный state).

**Решение:**
- Подключить LiveKit Data Channel для обмена сообщениями
- Синхронизация с веб-чатом платформы (зрители в браузере видят сообщения ведущего и наоборот)
- Протокол: JSON через Data Channel (type: "chat", sender, text, timestamp)

### 3.2.2. Функции чата

- Приём/отправка текстовых сообщений
- Модерация: бан/мьют участника (ведущий)
- Закреплённое сообщение
- Реакции (эмодзи)
- Уведомления о входе/выходе участников
- Счётчик онлайн-зрителей

**Критерий готовности:**
- Ведущий в приложении и зрители в браузере видят одни и те же сообщения
- Модерация работает из приложения

## 3.3. Автообновление

### 3.3.1. Tauri Updater Plugin

**Текущее состояние:** Проверка обновлений через GitHub Releases API, ручное скачивание.

**Решение:**
- Подключить `tauri-plugin-updater` (Tauri 2)
- Endpoint: GitHub Releases (уже используется)
- Flow: проверка при запуске → уведомление → скачивание в фоне → установка при перезапуске
- Пользователь видит прогресс скачивания
- Кнопка "Обновить и перезапустить"

### 3.3.2. Требования

- На macOS: подпись приложения (см. 3.5)
- На Windows: подпись через Authenticode (опционально, но убирает SmartScreen warning)
- Fallback: если автообновление не сработало — кнопка "Скачать вручную"
- Changelog в уведомлении об обновлении

**Критерий готовности:**
- Пользователь получает уведомление о новой версии
- Обновление устанавливается без ручного скачивания DMG
- Не нужна команда `xattr -cr` после обновления

## 3.4. Аппаратное кодирование

### 3.4.1. Зачем

Программное кодирование (libx264) нагружает CPU на 30-60%. Аппаратное — на 2-5%.

### 3.4.2. Поддерживаемые кодеки

| Платформа | Технология | Кодек |
|-----------|-----------|-------|
| macOS | VideoToolbox | H.264, HEVC |
| Windows (Nvidia) | NVENC | H.264, HEVC |
| Windows (AMD) | AMF | H.264, HEVC |
| Windows (Intel) | QSV (Quick Sync) | H.264, HEVC |

### 3.4.3. Реализация

- Интегрировать `ffmpeg` с hw-accelerated encoding через Rust
- Библиотека: `gstreamer` (Rust bindings) или прямые вызовы `ffmpeg` CLI
- Автоопределение доступного энкодера при запуске
- Fallback на software encoding если hw недоступен
- Настройка в UI: выбор энкодера (авто / software / hardware)

### 3.4.4. Пайплайн

```
Camera/Screen → Rust native capture → HW Encoder → LiveKit publish
                                          │
                                     VideoToolbox (Mac)
                                     NVENC (Nvidia)
                                     QSV (Intel)
```

**Критерий готовности:**
- CPU usage при стриме 1080p30 < 10%
- Определение доступного энкодера автоматическое
- Качество картинки не хуже software encoding

## 3.5. Подпись и установщики

### 3.5.1. macOS

**Проблема:** Без подписи macOS показывает "Приложение от неизвестного разработчика" и требует `xattr -cr`.

**Решение:**
- Apple Developer Account ($99/год)
- Code Signing Certificate (Developer ID Application)
- Notarization через `xcrun notarytool`
- В GitHub Actions: `APPLE_CERTIFICATE`, `APPLE_CERTIFICATE_PASSWORD`, `APPLE_ID`, `APPLE_TEAM_ID`

**Результат:** Пользователь открывает DMG → перетаскивает → работает. Без терминала.

### 3.5.2. Windows

- Authenticode подпись (опционально, ~$200-400/год через SSL.com или Sectigo)
- NSIS installer уже настроен
- Без подписи: SmartScreen warning при первом запуске (можно обойти)

### 3.5.3. Установщики

| Платформа | Формат | Автообновление |
|-----------|--------|----------------|
| macOS ARM | .dmg | Tauri Updater |
| macOS Intel | .dmg | Tauri Updater |
| Windows x64 | .exe (NSIS) | Tauri Updater |

## Порядок реализации

### Этап A — Нативный захват медиа (1-2 недели)
1. Rust: нативная камера через `nokhwa` или AVFoundation
2. Rust: нативный захват экрана через ScreenCaptureKit / DXGI
3. Фронтенд: получение видеопотока через Tauri IPC
4. Тестирование на macOS + Windows

### Этап B — Реальный чат + стабильность (1 неделя)
1. LiveKit Data Channel интеграция
2. Синхронизация с веб-чатом
3. Автореконнект при обрыве
4. Индикатор качества соединения

### Этап C — Автообновление (3-5 дней)
1. `tauri-plugin-updater` интеграция
2. UI для обновления (прогресс, changelog)
3. Тестирование на обеих платформах

### Этап D — Аппаратное кодирование (1-2 недели)
1. Интеграция VideoToolbox (macOS)
2. Интеграция NVENC/QSV/AMF (Windows)
3. Автодетект доступного энкодера
4. UI: выбор энкодера в настройках

### Этап E — Подпись и дистрибуция (3-5 дней)
1. Apple Developer Account
2. Настройка подписи в GitHub Actions
3. Notarization
4. Тестирование чистой установки
